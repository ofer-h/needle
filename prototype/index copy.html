Needle project handbook

A calm transition coach for the moment focus usually slips.

Needle catches the switch between task and meeting, asks for a quick brain dump, and helps the user start the next thing clean. This page summarizes what exists, what is planned, and why the architecture is shaped the way it is.

Current app macOS Electron, local-first
Shared spine pnpm, Turborepo, domain, contract
Product wedge intentional transitions, not task storage
01 / Overview

What Needle Is

Needle is a transition coach for a software engineer with ADHD or anyone who loses focus when moving from one thing to the next. It works at the boundary: one meeting ends, another task starts, but the last context is still stuck in the user's head. Needle interrupts at that hard stop, makes the user dump the residue quickly, and points attention toward the next thing.

For

Someone who benefits from external structure, low-friction capture, and gentle focus support during context switches.

Not

Not another to-do app, not a timer, and not a planner that only helps before the day starts.

Differentiator

It uses a well-timed interruption on purpose, and the long-term product thesis is to rotate the intervention so it does not become wallpaper.

Positioning line: every other app helps you plan. Needle helps you let go of one task and start the next, which is the exact moment focus is lost.

02 / Architecture

Two Paths, One Shared Core

The repo is now a pnpm + Turborepo monorepo. The current app lives in apps/desktop. Shared packages live in packages/domain, packages/contract, and packages/config. Planned packages include ui-web, design-tokens, api-client, db-schema, and ai.

Path A: macOS local-first

The Electron app keeps shipping on local SQLite. It remains usable offline and is never blocked by the future backend.

Path B: cloud later

The planned cloud track adds apps/server, apps/web, auth, Postgres, and later mobile. It uses the same domain and contract packages.

Anti-divergence guardrail: Mac SQLite and future Postgres are two storage representations of one model. The shared domain and contract packages are the spine that keep "connect later" from becoming a rewrite.

The desktop renderer already talks to main through the preload-exposed window.api. That IPC boundary is the connect seam: later, handlers can call a remote API plus local cache behind the same renderer surface.

03 / Today Scheduling

The Today Model

Needle separates the identity of work from its placement in the day. The implemented scheduling model distinguishes fixed anchors from flexible work and renders them as one interleaved timeline.

Fixed

Fixed items have a wall-clock start time and cannot be freely dragged. Calendar meetings are event-only and non-checkable. Fixed anchors divide the day into slots.

Flexible

Flexible tasks carry slotIndex and slotOrder. They float between anchors and can be reordered using fractional indexing.

Build the interleaved list.

buildTimeline() merges events, fixed tasks, and flexible tasks into a single Today sequence.

Drop into gaps, not onto rows.

Drag-and-drop uses useDroppable gap zones between items plus a DragOverlay clone. This avoids the element-swap behavior that made earlier SortableContext attempts feel wrong.

Let overdue return to Today.

Overdue and Today now share one DnD context so overdue flexible work can be moved back into the day's plan.

Move toward v2 plans.

The v2 model replaces UI-shaped fields with Item, ItemPlan, ItemOccurrence, and ItemRelation(type='contains'). View labels like dateLabel and isOverdue are derived, not durable truth.

04 / Torch

How the Torch Intervention Works

Torch is the current high-intensity transition intervention: a full-screen, multi-display attention takeover for unmissable commitments. It intentionally makes the screen harder to ignore while leaving the user with two humane exits: brain dump or skip with reason.

Scheduling selects an active intervention.

InterventionLayer reads v2 store interventions for the current actor. Due scheduled interventions are activated, then the highest-intensity surfacing intervention becomes the top one. If its strategy is attention_takeover_torch, the renderer sends torch:show.

Main opens overlays on every display.

showTorch() creates one frameless, transparent, always-on-top TorchWindow per display using screen.getAllDisplays(). The windows load the renderer with ?mode=torch, which forces data-theme="dark" for correct token resolution.

The screen dims around the cursor.

Main polls the absolute cursor position about 30 times per second and broadcasts torch:cursor. Each overlay renders a spotlight only if the cursor is on its display. The backdrop ramps from a light tint to a 0.95 opacity ceiling with up to 12px blur, using an ease-out escalation curve.

Each overlay embeds the TorchBanner.

The old separate hero banner window was replaced by an embedded 48px strip at the top of every overlay. It displays title · in Xm · HH:MM when a meeting start time is known, plus Brain dump and Skip....

Click-through stays mostly enabled.

Overlays normally call setIgnoreMouseEvents(true, { forward: true }). Cursor polling enables clicks only when the cursor is over the banner; full panel modes make all overlays interactive. The contract also exposes torch:set-interactive for the Electron forward-event pattern.

Skip requires a deliberate countdown.

torch:skip-init switches overlays into skip mode via torch:hero { mode: 'skip' }. SkipPanel asks for a meeting or task reason, supports optional notes for "Other", and requires a 4-second countdown. Confirm emits torch:skip-confirm, then main relays torch:closed { reason: 'skipped' } to the main app and hides Torch.

Brain dump saves transition residue.

torch:brain-dump-init shows BrainDumpPanel on all overlays. The textarea supports Cmd+Enter to save. Submit emits torch:brain-dump-submit, main relays torch:closed with brainDumpText, and InterventionLayer stores it as a capture entry through the v2 store and db:add-capture.

torch:show torch:payload torch:cursor torch:hero torch:skip-init torch:skip-confirm torch:brain-dump-init torch:brain-dump-submit torch:closed db:add-capture

Important honesty note: the source has a native critical notification and snooze plumbing, but the current embedded banner flow centers on Brain dump and Skip. The handbook treats the richer cross-device escalation chain as v2 architecture, not shipped behavior.

05 / Async UX + Observability

No More Infinite Dots

The async and observability docs make one rule explicit: user-triggered async work must end in success, actionable error, or cancel, with a hard timeout. This was added because endless "classifying" dots are trust-breaking in a companion app.

Async primitives

usePendingOperation models idle, pending, success, and error. It defaults to a 30s timeout, ignores stale IPC results after cancel or a newer run, and marks flows slow after 3s.

Visible pending UI

AsyncStatusPanel shows elapsed seconds, a slow hint, and cancel action. Capture classify and API key save are the examples already called out in the docs.

Local logs

Main logs use [needle]. Renderer logs use [needle-ui]. Safe metadata includes duration, outcome, key source, text length, and flow IDs, never secrets or raw brain-dump text.

Flow health

The main process keeps a last-50 event ring buffer exposed through app:getFlowHealth. BuildDiagnostics shows version, SHA, API key source, and last classify status in dev.

Still open from the docs: Today hydrate pending UI, silent DB persist feedback, fire-and-forget capture persistence, packaged main logs, and passing flowId back to renderer consistently.

08 / Suggestions

Grounded Next Steps

These are suggestions synthesized from the docs' next-step sections and open gaps. They are not claims that the work already exists.

  • Close the remaining async/observability P0s. Add Today hydrate pending UI, visible DB persist feedback, safer fire-and-forget capture persistence, packaged logging strategy, and renderer flowId propagation.
  • Run and document the final monorepo smoke test. Confirm app launch, SQLite read/write persistence, Capture classify, and root pnpm run typecheck plus pnpm run lint from the current branch.
  • Kick off Track B deliberately. Scaffold apps/server first with NestJS, Postgres, better-auth, AccessModule, and ts-rest contract implementation, or run server and web agents in parallel against @needle/contract mocks.
  • Extract ui-web only when web starts. Let apps/web drive the transport-agnostic boundary instead of guessing it from desktop alone.
  • Keep strengthening the v2 Today adapter. Continue moving away from UI-shaped state toward Item, ItemPlan, ItemOccurrence, and child item relations.
  • Instrument Torch outcomes. The data model has interventions, transition events, and capture entries; next useful product learning is whether brain dump, skip, timeout, or escalation actually helps transitions.
  • Prototype current focus and reflection. The roadmap calls for a current focus affordance, lightweight completion reflection, and daily flow state before deeper AI automation.
  • Keep accountability private-first. The scoring/coaching idea is captured but still conceptual; start with self-accountability and local adherence metrics before coach or leaderboard surfaces.
