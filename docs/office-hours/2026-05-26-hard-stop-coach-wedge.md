## Problem Statement

People with ADHD lose time and energy at every **hard stop** in their day — the moment when one focus mode has to end and another has to begin. A "hard stop" is a product noun. It can be triggered by:

- A calendar meeting starting (externally enforced).
- A lunch break (time-of-day, user-authored).
- A scheduled check-in: *"am I still on the most important thing?"*
- An on-track validation: *"is this work actually moving the needle?"*
- Future: AI-detected drift, end-of-deep-work-block, location change.

The cognitive cost of letting-go-and-restarting is the load-bearing pain, not "I can't organize my tasks" (which is the saturated ADHD-app market thesis). Existing productivity tools treat the problem as an organization problem; the actual bottleneck is neurochemistry — habituation, novelty hunger, and the executive function of mental closure.

Needle's wedge is to be the **rotating kit of attention interventions at hard stops**. The kit is two-dimensional: **trigger sources × intervention strategies.** Variety along both axes is what defeats habituation.

## Target User & Narrowest Wedge

**Target user:** Anyone whose workday involves shifting focus and who benefits from intentional hard stops. This is broader than "calendar-heavy operators" — it includes ICs with one meeting a day, researchers with no meetings, and anyone who's spent 2+ years working with an ADHD coach on day-planning discipline. **What unifies them is not meeting density — it's the desire for externally-prompted hard stops** as a way to interrupt drift, hyperfocus, or context-rot.

In v1, the named warm leads are:
- **Alon** — VP Product, AI/insurance. Calendar-heavy. Validates the meeting-transition trigger.
- **Founder (Ofer)** — 2.5y ADHD-coaching practice. Validates the user-authored hard-stop (lunch, on-track, validation) triggers.

These two together cover the two axis ends of the target user. If both engage, the breadth thesis holds.

**Narrowest wedge — "Hard Stop Coach":**

In v1, **three trigger sources** ship together (not just calendar):
1. **Calendar trigger** — connects to macOS Calendar via EventKit. 5 min before each meeting.
2. **User-authored time-of-day trigger** — user can set "every weekday at 12:30, hard-stop for lunch" or "every 90 min between 9am-5pm, hard-stop to validate I'm on track." Lightweight scheduling UI: pick time(s), pick intervention strategy, save.
3. **Manual "check-in now" trigger** — global keyboard shortcut (`⌘⇧K`) fires an immediate hard-stop intervention. For the moment the user realizes they've drifted.

**Two intervention strategies from day 1** (the strategy axis of the kit thesis):
1. **Torch** — full-screen blur/dim with cursor spotlight + embedded brain-dump panel. For high-importance hard stops (`unmissable` calendar events; user-flagged "this matters").
2. **Ambient pill** — quieter top-right notification badge + click-to-open brain-dump. For lower-importance hard stops (`firm` calendar events; routine lunch check-ins).

**Brain-dump panel:** prompt is **context-aware** based on trigger type:
- Calendar trigger: "What do you need to let go of?"
- Lunch trigger: "What's been the morning's biggest win? What's the afternoon's one priority?"
- Check-in trigger: "Are you still on the most important thing?"
- Manual trigger: "What just happened in your head?"

**Storage:** writes to local SQLite (instrumentation + restart-resilience), but the UI does not surface dumps anywhere in v1 — no inbox, no search, no Today-screen entries. The *act* of dumping is 90% of the value (premise #9); the row is for instrumentation and a possible week-14 retrieval UX.

**Escalation rule (per-strategy, rolling window):** if a strategy is dismissed in under 2 seconds three times within a rolling 7-day window for a given user, the system promotes to the next strategy on the ladder. Ladder for week 1: `ambient_pill → torch → escalated_alert (macOS critical notification)`. Resets if user engages (brain-dump completed or >2s acknowledgement). Novelty defeats habituation.

**Commitment level:** every calendar event defaults to `firm` (→ ambient_pill). User can manually upgrade an event to `unmissable` via a right-click context action in the event row, **shipped in week 1** (without this, the torch has no path to fire on calendar triggers). User-authored time-of-day triggers default to `unmissable` at first because the user just explicitly chose to schedule them — but the user can override that during creation.
- Instrumentation from day 1: log every intervention trigger, dismiss time, completion, skip.
- **$20/mo charged on day 1 of use** — commitment device, not revenue.

**Cut from the wedge (deferred to v2 — code is kept, just not shipped as primary UI):** Today screen, Remember/Act buckets, Capture screen as standalone, AI classification, subtasks UI polish, Capture inbox, semantic search, multi-app, mobile, web, NestJS backend, sync, monorepo. Today screen code stays in the repo as a debug/admin view behind a hidden settings flag.

**Explicit non-trigger sources in v1 (deferred):** AI-detected drift, end-of-deep-work-block detection (would require focus tracking), location-based triggers (would require accessibility permissions), wearable triggers (would require BLE pairing). These all live in the future product but are out of v1 scope.

## Approaches Considered

### Approach A — Subagent's 48-Hour Ship
Torch + brain-dump + EventKit + local SQLite + signed .dmg. One intervention type. $20/mo day 1.
- Pros: Fastest learning loop; reuses ~all built code; forced shipping discipline.
- Cons: Doesn't test the kit thesis (premise #8); risk of day-10 abandonment.

### Approach B — Kit Foundation **(CHOSEN, then expanded post-review)**
A + ambient_pill (second strategy) + user-authored time-of-day triggers + manual `⌘⇧K` trigger + escalation rule + instrumentation. Tests the *full* two-dimensional kit thesis (trigger sources × strategies) on day 1. ~10-14 days.
- Pros: Proves the actual thesis from day 1; differentiates vs Saner.AI/Pith; instrumented for churn detection; addresses the wider target user.
- Cons: 10-14 days instead of 7; ambient pill + scheduling UI both need design passes.

### Approach C — Daily Ritual
A + B + morning kickoff intervention + end-of-day wind-down. Three time-anchored interventions. ~2 weeks.
- Pros: Sharpest pitch ("Needle runs your day"); aligns to v2 product direction exactly.
- Cons: Two new intervention strategies to design; likely slips from 2 to 4 weeks; risk Alon says "too much."

## Recommended Approach: B — Kit Foundation

Rationale:
- Approach A is too narrow to test the *actual* product thesis (premise #8).
- Approach C is too wide to validate the wedge in 30 days.
- B proves the kit thesis with the smallest possible second-strategy addition, ships in a week, and instruments churn risk from day 1.
- B preserves optionality: if Alon engages, expanding to C is straightforward (the domain model already supports it).

  - **Offline behavior:** cache the last successful entitlement check locally with a 14-day TTL; if offline and cache valid, allow use; if offline and cache expired, allow use in degraded mode (interventions still fire, but a non-blocking banner says "verify your subscription when you're back online"). ## Dependencies

**Required for wedge ship (not built yet):**
- macOS EventKit integration. **Chosen approach: a thin Swift helper binary** that reads EventKit and writes JSON to stdout, spawned by the Electron main process via `child_process`. Reasons: (1) no npm packages currently target EventKit reliably on Apple Silicon + modern macOS, (2) the Swift helper is ~80 lines and dodges native-module rebuild pain on Electron version bumps, (3) easy to harden permissions via Info.plist. **Day-1 spike risk:** this is the single highest-uncertainty piece in the build; if the Swift helper costs more than 2 days, fall back to Apple Script + `osascript` reading from Calendar.app — slower but no native compile.
- Local SQLite via `better-sqlite3` (already in the broader plan, not yet implemented).
- Stripe entitlement check (cheap serverless function).
- Instrumentation table + writer in SQLite for intervention events. Schema sketch: `(id, ts, strategy, surface, intensity, trigger_source, dismissed_within_ms, brain_dump_completed)`.
- **Dismissal counter (escalation persistence):** either a derived query over the instrumentation table (cheap if events are few), or a separate small `dismissal_counters(user_id, strategy, count, window_start_ts)` table updated on each dismiss. Pick whichever ships faster; both work for v1.
- Ambient pill intervention surface (new UX).
- Escalation logic ("dismissed <2s × 3 → switch strategy") in the InterventionLayer.
- **User-authored hard-stop scheduling UI** — pick time(s), pick recurrence (daily / weekday / once), pick strategy, pick brain-dump prompt template (or use default). Designed to be 30-second creation, not a settings page.
- **Manual hard-stop shortcut** — `⌘⇧K` global handler in main process; fires `InterventionLayer.fireManualHardStop()`. Uses the cheapest strategy (ambient_pill) so it doesn't punish the user for triggering themselves.
- **Context-aware brain-dump prompts** — small lookup table keyed on trigger type. Hard-coded for v1; user-editable later.
- Signed .dmg build configured in Electron Forge.
